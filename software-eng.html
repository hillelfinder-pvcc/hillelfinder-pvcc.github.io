Chapter 7: Software Engineering

    Software engineering
  • The branch of computer science that seeks principles to guide the development of large, complex software systems
    Integrated development environment (IDE)
  • A system which combines tools for developing software, such as editors compilers, and debugging tools

    Software life cycle: development phase
    1. Requirements analysis
      • Specifies what services the proposed system will provide, to identify any conditions (time constraints, security, etc) on those services, and to define how the outside world will interact with the system
    2. Design
      • Creating a plan for the construction of the proposed system
    3. Implementation
      • The actual writing of programs, creation of data-files, and the development of databases for the system
    4. Testing
      • The debugging of programs, as well as ensuring that each step of the development process is checked for accuracy
    Modular implementation
  • The division of software into manageable units, each of which deals with only a part of the overall responsibility
    Unified modeling language (UML)
  • The modern collection of tools with the purpose promoting object-oriented programming (see chapter 6)
    Use case
  • Interactions between the system and its users
    Use case diagram
  • A diagram which depicts the proposed system, the actors who will use it, and its use cases
    Data relationships
    1. One-to-one
      • A situation of relationships in which a user is associated with one and only one piece of data, and that piece of data is also associated with only that one user
    2. One-to-many
      • A situation of relationships in which a user is associated with multiple pieces of data, but that piece of data is also associated with only that one user
    3. Many-to-many
      • A situation of relationships in which a user is associated with multiple pieces of data, and that piece of data is also associated with multiple users
    Design pattern
  • A predeveloped model for solving a recurring problem in software design
    Software quality assurance (SQA)
  • The process of overseeing and enforcing the quality control system adopted by the organization creating the software
    Software testing:
    1. Glass box testing
      • The software tester is aware of the interior structure of the software and uses this knowledge when designing the test
    2. Black box testing
      • Tests that do not rely on knowledge of the software's interior composition (from the perspective of a user, rather than a developer)
    3. Beta testing
      • A type of black-box testing in which a preliminary version of the software is given to a segment of the intended audience with the goal of learning how the software is given to a segment of the intended audience with the goal of learning how the software performs in real-life situations before the final version of the product is solidified and released to the market
    4. Alpha testing
      • Similar to beta-testing, but performed at the developer's site
    Documentation
    1. User documentation
      • To explain features of the software and to describe how to use them. It is intended to be read by the use of the software and is therefore expressed in the terminology of the application
    2. System documentation
      • To describe the software’s internal composition so that the software can be maintained later in its life cycle; the source version of all the programs in the system
    3. Technical documentation
      • To describe how a software system should be installed and serviced (such as adjusting operating parameters, installing updates, and reporting problems back to the software’s developers
    Intellectual property law
    1. Software license
      • The rights of the developer expressed in legal terms. It is the legal agreement between the owner and use of the a software product that grant the user certain permissions to use the product without transferring ownership rights to thee intellectual property
    2. Software disclaimer
      • A disclaimer in a software license that states the limitations of the liability of the company producing the software
    3. Best defenses against software liability claims
      • "Consequently, one of the best defenses against software liability claims is to apply sound software engineering principles during the software;s development, to use a level of care compatible with software’s application, and to produce and maintain records that validate these endeavors"
